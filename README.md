# EduRank: Adaptive Bradley-Terry Ranking

This project uses an **adaptive Bradley-Terry (BT) pairwise ranking pipeline** to estimate item quality from accumulated pairwise comparisons.

## Overview

The system ranks videos within each topic based on pairwise match results. Instead of relying on a fixed comparison schedule, it repeatedly:

1. collects pairwise outcomes,
2. fits BT scores from all observed comparisons,
3. updates the topic ranking,
4. selects the next most informative pairs to compare.

This creates a closed ranking loop in which later comparisons are chosen based on the current model state, allowing the system to focus labeling effort on the matchups that are most useful for improving the ranking.

## Ranking Pipeline

### 1. Store pairwise outcomes

Each comparison result is stored in `db.matches` as:

```ts
(winnerId, loserId)
```

Each record indicates that `winnerId` was preferred over `loserId`.

### 2. Fit Bradley-Terry scores

For each topic, the system fits **Bradley-Terry scores** using all accumulated pairwise outcomes.

These scores represent the relative strength of each video under the BT model.

### 3. Produce the ranking

The topic ranking is generated by sorting videos by their fitted `btScore`.

A higher `btScore` indicates a stronger item and therefore a higher rank.

### 4. Select the next comparison pairs adaptively

Instead of sampling pairs uniformly at random, the system selects the next batch of comparisons using an adaptive scoring rule.

By default, the next **8 pairs** are chosen based on a combination of the following factors:

- **Uncertainty**  
  Prefer pairs whose predicted outcome is close to 50/50:

  ```ts
  p = sigmoid(s_i - s_j)
  uncertainty = p * (1 - p)
  ```

  This favors comparisons that are most informative for refining the ranking.

- **Coverage bonus**  
  Prefer items that have been compared fewer times:

  ```ts
  1 / (count_i + count_j + 1)
  ```

  This helps prevent some items from being under-sampled.

- **Repetition penalty**  
  Downweight pairs that have already been shown too many times:

  ```ts
  1 / (1 + pair_repeat_count)
  ```

  This reduces redundant comparisons.

- **Optional rank-neighbor bonus**  
  Optionally increase the score of pairs that are currently close in rank.

  This can help refine local ordering, especially near ranking boundaries. The bonus is damped during the low-data phase so that early exploration remains broad.

## Recommended Default Settings

These defaults are a good starting point for topics with around **16 items**:

```ts
adaptiveInitialMinComparisons = 2
adaptiveMaxPairRepeats = 3
adaptiveAdjacentBonus = 0.35
adaptiveBatchSize = 8
```

## Parameter Guide

### `adaptiveInitialMinComparisons = 2`

Ensures that each item receives at least a small number of comparisons before the adaptive policy becomes too focused.

Increase this value if early rankings are noisy or unstable.

### `adaptiveMaxPairRepeats = 3`

Limits how many times the same pair can be shown.

Decrease this value if users are seeing repeated matchups too often.

### `adaptiveAdjacentBonus = 0.35`

Encourages comparisons between items that are currently close in rank.

Decrease this value if pair selection becomes too locally focused and stops exploring the wider pool.

### `adaptiveBatchSize = 8`

Controls how many pairs are selected per round.

Increase this value if your labeling workflow benefits from larger batches and fewer ranking-update cycles.

## When To Tune

You may want to adjust the defaults depending on dataset size, judgment noise, and annotation budget.

- Increase `adaptiveInitialMinComparisons` if early rankings are unstable.
- Decrease `adaptiveMaxPairRepeats` if repeated matchups become too frequent.
- Decrease `adaptiveAdjacentBonus` if selection over-focuses on nearby ranks too early.
- Increase `adaptiveBatchSize` if your workflow benefits from larger labeling rounds.

## Notes on Legacy Fields

- `btScore` is the primary ranking signal and the source of truth.
- The legacy `elo` field is retained only for compatibility or display purposes.
- `elo` is derived from BT and should not be treated as the authoritative ranking value.

## Summary

EduRank now uses:

- **Bradley-Terry fitting** for score estimation,
- **adaptive pair selection** for efficient data collection,
- and **`btScore`-based sorting** for final ranking.

This makes the ranking process more data-efficient than fixed or random pairing, especially when the number of items per topic is moderate and labeling budget is limited.
